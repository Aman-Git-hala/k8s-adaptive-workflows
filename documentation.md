## 1. What This Project Does

K8s Adaptive Workflows is a **Kubernetes Operator** that runs DAG-based (Directed Acyclic Graph) workflows inside a Kubernetes cluster, but unlike standard job runners (Argo, plain Jobs), it is **adaptive**: it uses a machine learning Inference Engine to *predict* how many CPU/memory resources each task will need, and a smart Optimizer to *schedule* tasks in an order that stays within cluster resource limits.

**The core innovation:** Instead of blindly launching all ready tasks at once (which can exceed cluster memory and get pods evicted), the operator predicts resource needs and bins tasks together like a 2D Tetris game — sliding tasks along the time axis when they'd exceed memory limits — maximizing throughput without crashes.

**Data flow per reconciliation cycle:**

```

    │

    ├─ 2. Handle finalizer (cleanup on delete)
    ├─ 3. Initialize TaskStatuses (Pending) if new
    ├─ 4. syncPodsToStatus() → read actual Pod phases
    ├─ 5. Check for failures or completion
    ├─ 6. Inference.Predict() → per-task CPU/Mem/Duration estimates
    ├─ 7. Optimizer.Plan()   → which tasks to start NOW and with what resources
    ├─ 8. createTaskPod()    → spawn Pods for each scheduled task
    ├─ 9. Update CR Status
    └─ 10. Requeue after 10s
```

## 4. CRD — The Custom Resource Definition

**Location:** `api/v1/`

This is what users write to describe a workflow. It gets applied to Kubernetes as a YAML file.

### `adaptiveworkflow_types.go`

Defines the schema for the `AdaptiveWorkflow` Custom Resource. This is the most important type file.

#### `TaskTemplate` — one task in the DAG

```go
type TaskTemplate struct {
    Name         string                      // Unique task identifier
    Image        string                      // Docker image to run
    Command      []string                    // Shell command (optional)
    Dependencies []string                    // Names of tasks that must finish first
    BaseResources corev1.ResourceRequirements // Optional hint for the Inference Engine
}
```

- **`Dependencies`** defines the DAG edges. If `task-B` depends on `task-A`, the controller will not start `task-B` until `task-A` succeeds.
- **`BaseResources`** is an *optional hint* that the user provides. The Baseline Inference Engine will pass it through as-is; the ML Inference Engine may override it with better predictions from historical data.

#### `AdaptiveWorkflowSpec` — the full workflow definition

```go
type AdaptiveWorkflowSpec struct {
    Tasks           []TaskTemplate        // All tasks in the DAG
    MaxResources    corev1.ResourceList   // Hard ceiling on concurrent resource usage
    OptimizationGoal string              // "MinimizeTime" (default) or "MinimizeCost"
}
```

- **`MaxResources`** tells the Optimizer the absolute cluster budget. It will never schedule a combination of tasks that exceeds this.
- **`OptimizationGoal`** is currently used as a hint; defaults to `MinimizeTime`.

#### `AdaptiveWorkflowStatus` — live state (updated by the controller)

```go
type AdaptiveWorkflowStatus struct {
    Phase               WorkflowPhase             // Pending / Running / Completed / Failed
    TaskStatuses        map[string]TaskStatus     // Per-task live state
    CurrentResourceUsage corev1.ResourceList      // Sum of all Running tasks' allocations
    Conditions          []metav1.Condition        // Standard K8s conditions (Progressing, Available)
}
```

#### Task/Workflow Phase enums

| Constant | Meaning |
|----------|---------|
| `TaskPhasePending` | Task is waiting for dependencies |
| `TaskPhaseRunning` | Pod is running |
| `TaskPhaseSucceeded` | Pod finished successfully |
| `TaskPhaseFailed` | Pod failed |
| `WorkflowPhasePending` | Workflow just created |
| `WorkflowPhaseRunning` | At least one task is active |
| `WorkflowPhaseCompleted` | All tasks succeeded |
| `WorkflowPhaseFailed` | Any task failed |

### `groupversion_info.go`

Sets up the Kubernetes API registration. Defines:
- **Group:** `v1.wannabe.dev`
- **Version:** `v1`
- **Kind:** `AdaptiveWorkflow`

The `SchemeBuilder` variable registers these types with the Kubernetes scheme so controller-runtime knows how to serialize/deserialize them.

### `zz_generated.deepcopy.go`

**Auto-generated. Do not edit.** Contains `DeepCopyObject()` and `DeepCopyInto()` methods. Required by Kubernetes for all API types. Regenerate with `make generate`.

## 5. Go Controller (The Brain)

**Location:** `internal/controller/adaptiveworkflow_controller.go`

This is the reconciliation loop — the heart of the operator.

### `AdaptiveWorkflowReconciler` struct

```go
type AdaptiveWorkflowReconciler struct {
    client.Client            // Kubernetes API client
    Scheme    *runtime.Scheme
    Inference inference.Engine      // Pluggable: baseline Go or gRPC Python
    Optimizer optimizer.Optimizer   // Pluggable: greedy Go or gRPC C++
    StateDB   statedb.StateDB       // Pluggable: PostgreSQL or no-op
    Recorder  record.EventRecorder  // Kubernetes Events
}
```

All three service dependencies (`Inference`, `Optimizer`, `StateDB`) are **interfaces**, so they can be swapped without changing the controller. By default the in-process Go implementations are used; if the corresponding env vars are set, gRPC clients are injected instead.

### `Reconcile()` — step by step

**Step 1: Fetch the CR**  
Uses `r.Get(ctx, req.NamespacedName, wf)`. If `NotFound`, the object was deleted — return cleanly.

**Step 2: Finalizer handling**  
Before doing any work, checks if the object is being deleted (`DeletionTimestamp != nil`). If so, runs cleanup (currently removes the finalizer, which allows Kubernetes to garbage-collect owned pods). Adds the finalizer `v1.wannabe.dev/finalizer` on first reconcile so we always get a chance to clean up.

**Step 3: Initialize TaskStatuses**  
On the very first reconcile (when `Status.TaskStatuses == nil`), populates all tasks as `Pending` and sets workflow phase to `Pending`. Emits a Kubernetes Event.

**Step 4: Skip terminal workflows**  
If the workflow is already `Completed` or `Failed`, returns immediately (no requeue).

**Step 5: `syncPodsToStatus()`**  
Lists all Pods owned by this workflow (via labels `adaptive-workflow=<name>` and `task-name=<task>`). For each Running pod that has transitioned to `Succeeded` or `Failed`, updates the corresponding `TaskStatus.Phase`. When a task succeeds, it also calls `StateDB.RecordExecution()` to log the actual duration and resource usage for future ML training.

**Step 6: Check for workflow failure**  
If any task is `Failed`, marks the whole workflow `Failed` and returns.

**Step 7: Check for completion**  
If all tasks are `Succeeded`, marks the workflow `Completed` and returns.

**Step 8: DAG Execution (Inference → Optimizer → Pods)**
1. **`r.Inference.Predict(wf.Spec)`** — asks the Inference Engine for per-task resource predictions
2. **`r.Optimizer.Plan(spec, taskStatuses, predictions)`** — asks the Optimizer which tasks to launch now (respecting DAG order and resource constraints)
3. For each `TaskAllocation` in the plan, calls **`r.createTaskPod()`**

**Step 9: Update resource usage**  
Sums up resource requests of all currently Running tasks and stores in `CurrentResourceUsage`.

**Step 10: Update status and requeue**  
Calls `r.Status().Update()` then requeues after 10 seconds (`requeueInterval`).

### `createTaskPod()`

Creates a standard `corev1.Pod` with:
- Labels: `adaptive-workflow`, `task-name`, `app.kubernetes.io/managed-by`
- `RestartPolicy: Never` (tasks run once)
- Resources from the Optimizer's allocation
- Owner reference pointing to the `AdaptiveWorkflow` (so pods are auto-deleted when the workflow is deleted)

Checks if a pod already exists before creating — idempotent.

### `syncPodsToStatus()`

Lists pods by label first. Falls back to field-index query if available. For each relevant pod, maps Kubernetes `PodPhase` → internal `TaskPhase`.

### `computeResourceUsage()`

Iterates Running tasks and sums their CPU and memory requests. Returns a `ResourceList`.

### `podName(workflowName, taskName)`

Generates deterministic pod names: `<workflow-name>-<task-name>` (lowercased). This is why names must be RFC 1123 compliant.

### Controller Tests

**`adaptiveworkflow_controller_test.go`** uses Ginkgo + Gomega with a real (envtest-backed) Kubernetes API server:
1. Creates an `AdaptiveWorkflow` with 2 tasks
2. Asserts both tasks transition to `Running` (pods spawned)
3. Simulates pod success by updating their phase to `Succeeded`
4. Asserts the workflow transitions to `Completed`

**`mocks_test.go`** provides mock `Optimizer` and `Inference` implementations used in tests.

**`suite_test.go`** bootstraps the envtest environment (real k8s APIserver + etcd) and registers the controller with a test manager.

## 6. Internal Packages

### 6.1 `internal/inference`

#### `inference.go` — The Interface

```go
type Engine interface {
    Predict(spec v1.AdaptiveWorkflowSpec) (optimizer.Predictions, error)
}
```

One method: takes the full workflow spec, returns a map of `taskName → TaskPrediction`.

#### `baseline.go` — The Default In-Process Implementation

`BaselineEngine` is the simplest possible Inference Engine. It:
- Reads each task's `BaseResources` field from the spec
- Returns it verbatim as the prediction
- If no `BaseResources` are set, returns no prediction (Optimizer falls back to defaults)

**When is it used?** When `INFERENCE_GRPC_ADDR` is not set. It's the zero-dependency fallback.

### 6.2 `internal/optimizer`

#### `optimizer.go` — Types and Interface

Key types:

```go
type TaskPrediction struct {
    EstimatedDuration  float64                    // seconds
    EstimatedResources corev1.ResourceRequirements // CPU + Memory
}

type Predictions map[string]TaskPrediction  // taskName → prediction

type TaskAllocation struct {
    TaskName  string
    Resources corev1.ResourceRequirements  // what to actually set on the Pod
}

type SchedulePlan struct {
    TasksToStart []TaskAllocation  // tasks to launch this cycle
}

type Optimizer interface {
    Plan(spec, taskStatuses, predictions) (SchedulePlan, error)
}
```

#### `greedy.go` — The Default In-Process Implementation

`GreedyOptimizer` implements a resource-constrained greedy scheduling algorithm:

1. **Calculate current usage:** Sum CPU and memory of all Running tasks
2. **For each Pending task (in spec order):**
   - Check all dependencies are `Succeeded` → skip if not ready
   - Determine resources: prediction → BaseResources → default (100m CPU, 128Mi Mem)
   - Check if adding this task would exceed `MaxResources` → skip if it would overflow
   - If it fits, add to the plan, update running totals
3. Return the plan

**Key behavior:** Tasks are checked in spec order. If a task would overflow the resource budget, it is skipped *for this cycle* — it will be scheduled in a future reconcile when other tasks finish and free up resources. This is the "Tetris" effect.

**When is it used?** When `OPTIMIZER_GRPC_ADDR` is not set.

### 6.3 `internal/grpcclient`

These files bridge the Go interfaces to remote gRPC microservices.

#### `inference_client.go` — Python Inference Engine Client

`InferenceClient` implements `inference.Engine` but sends all calls over gRPC to the Python server:
- Converts `v1.AdaptiveWorkflowSpec` → `pb.PredictRequest` (proto format)
- Calls `InferenceService.Predict()` over gRPC (30s timeout)
- Converts `pb.PredictResponse` → `optimizer.Predictions` (Go format)
- Uses 1.5× multiplier for limits vs requests (adds buffer headroom)

Also provides `ReportMetrics()` to send actual execution data back to the Python server.

#### `optimizer_client.go` — C++ Optimizer Client

`OptimizerClient` implements `optimizer.Optimizer` but sends all calls over gRPC to the C++ server:
- Converts `v1.AdaptiveWorkflowSpec` + `TaskStatuses` + `Predictions` → `pb.PlanRequest`
- Calls `OptimizerService.Plan()` over gRPC (30s timeout)
- Converts `pb.PlanResponse` → `optimizer.SchedulePlan` with `ResourceRequirements`

### 6.4 `internal/statedb`

#### `client.go` — PostgreSQL Execution History

The `StateDB` interface has three methods:

```go
type StateDB interface {
    RecordExecution(ctx, TaskExecution) error
    GetHistoryByImage(ctx, image) (*HistoricalStats, error)
    Close() error
}
```

**`PostgresStateDB`** — real PostgreSQL backend:
- Uses `database/sql` with `lib/pq` driver
- `RecordExecution()`: INSERT into `task_executions` (workflow, task, image, cpu_m, mem_mib, duration_s, succeeded, completed_at)
- `GetHistoryByImage()`: aggregated `AVG()` query over last 30 days of successful runs

**`NoOpStateDB`** — silent discard:
- Used when `STATEDB_CONNECTION_STR` env var is not set
- All operations succeed silently — operator still works, just no learning

#### `schema.sql`

```sql
CREATE TABLE IF NOT EXISTS task_executions (
    id            BIGSERIAL PRIMARY KEY,
    workflow_name TEXT NOT NULL,
    task_name     TEXT NOT NULL,
    image         TEXT NOT NULL,
    actual_cpu_m  BIGINT,
    actual_mem_mib BIGINT,
    duration_s    DOUBLE PRECISION,
    succeeded     BOOLEAN,
    completed_at  TIMESTAMPTZ DEFAULT NOW()
);
```

This is the **feedback loop table**. Every time a task finishes, the controller records the actual runtime and resource usage. The Python Inference Engine reads this table to train its ML model.

## 7. `cmd/main.go` — Manager Entry Point

The main entry point for the **controller manager** binary. Wired by Kubebuilder scaffold.

**What it does:**
1. Parses CLI flags (metrics addr, health probe addr, leader election, TLS certs for webhooks/metrics)
2. Creates a `controller-runtime Manager` — this manages the reconciler loop, scheme, and client
3. Decides which service implementations to inject, based on env vars:
   - **`INFERENCE_GRPC_ADDR`** set → `grpcclient.NewInferenceClient(addr)` (connects to Python). Else → `inference.NewBaselineEngine()`
   - **`OPTIMIZER_GRPC_ADDR`** set → `grpcclient.NewOptimizerClient(addr)` (connects to C++). Else → `optimizer.NewGreedyOptimizer()`
   - **`STATEDB_CONNECTION_STR`** set → `statedb.NewPostgresStateDB(connStr)`. Else → `statedb.NewNoOpStateDB()`
4. Creates and registers `AdaptiveWorkflowReconciler` with all injected deps
5. Adds health/ready probes (`/healthz`, `/readyz`)
6. Starts the manager (blocks until SIGTERM/SIGINT)

kwf submit <file.yaml>   Apply an AdaptiveWorkflow to the cluster

2. Decode as `unstructured.Unstructured` (to get name/namespace)

**`status` command:**  
Fetches the workflow and pretty-prints a table with each task's phase, pod name, and allocated CPU/memory.

**`list` command:**

### `ui.go` — REST API Proxy

When `kwf ui` runs, it starts an HTTP server on `:8080` that:

## 9. Python Inference Engine

**Location:** `inference-engine/`

A standalone Python gRPC microservice implementing the **ML prediction** brain.

### `server.py`

The gRPC server. Implements `InferenceService` (from `proto/inference.proto`):

#### `Predict()` — main prediction logic

For each task in the request, runs through a priority chain:

| Priority | Condition | Action |
|----------|-----------|--------|
| 1 (best) | ONNX model loaded + historical data exists | Run ML inference via ONNX Runtime |
| 2 | No model but have historical data | Use raw PostgreSQL averages + 20% headroom |
| 3 | No history but user provided BaseResources hint | Use user hint verbatim |
| 4 (fallback) | Nothing available | Use hardcoded defaults (100m CPU, 128Mi, 30s) |

Confidence scores are computed and included — higher samples = higher confidence (capped at 0.9).

#### `ReportMetrics()` — feedback loop

Accepts `TaskMetrics` from the controller and INSERTs them into PostgreSQL. This is how the system learns from actual execution data.

#### `_get_history()` — PostgreSQL query

```sql
SELECT AVG(actual_cpu_m), AVG(actual_mem_mib), AVG(duration_s), COUNT(*)
FROM task_executions
WHERE image = %s AND succeeded = TRUE AND completed_at > NOW() - INTERVAL '30 days'
```

Groups by container image — same image → same resource pattern assumption.

### `train_model.py`

Generates synthetic training data (various image types with different resource characteristics) and trains a `RandomForestRegressor` using scikit-learn to predict `[cpu_m, mem_mib, duration_s]` from `[image_hash, hist_cpu, hist_mem, hist_dur]`. Exports to `model.onnx` using `sklearn-onnx`. Run automatically during Docker build.

### `Dockerfile`

Multi-stage build: installs Python deps, runs `train_model.py` to bake a trained ONNX model into the image, then starts `server.py`.

### `requirements.txt`

```
grpcio, grpcio-tools
onnxruntime
scikit-learn, skl2onnx, numpy
psycopg2-binary
```

## 10. C++ Optimizer Service

**Location:** `optimizer-service/`

A standalone C++ gRPC microservice implementing the **scheduling** brain.

### `src/greedy_optimizer.h` + `greedy_optimizer.cpp`

The core scheduling algorithm implemented in C++. Mirrors the Go `GreedyOptimizer` but is fully standalone:

**`GreedyOptimizer::Plan()`:**
1. Parse task dependency graph
2. Track current resource usage from task statuses
3. For each Pending task: check if all deps succeeded
4. If ready: pick resource allocation (prediction → base resources → default)
5. Check if it fits within `max_resources` — skip if it would overflow
6. Add to result if fits; update running totals

**Internal types** (in `greedy_optimizer.h`):
- `DAGTask` — task definition with name, image, command, deps, optional base resources
- `TaskStatus` — current phase + allocated resources
- `TaskPrediction` — predicted CPU/mem/duration
- `PlanRequest` / `PlanResponse` — input/output contract

### `src/main.cpp`

The C++ gRPC server entry point:
- Reads `GRPC_PORT` env var (default `50052`)
- Enables gRPC server reflection (for `grpcurl` debugging)
- Implements `OptimizerService::Plan()` RPC
- Converts proto types → internal C++ types → calls `GreedyOptimizer::Plan()` → converts back to proto → returns

**Conversion helpers:**
- `ProtoToPlanRequest()` — proto → internal types
- `PlanResponseToProto()` — internal types → proto

Handles `SIGINT`/`SIGTERM` for graceful shutdown.

### `CMakeLists.txt`

CMake build script:
- Finds installed gRPC and Protobuf packages
- **Auto-generates** C++ gRPC stubs from `proto/optimizer.proto` at build time
- Compiles `main.cpp` + `greedy_optimizer.cpp` → `optimizer_server` binary

### `Dockerfile`

Multi-stage build:
1. **Build stage:** Ubuntu 22.04, installs cmake/gcc/libgrpc-dev, compiles the C++ binary
2. **Runtime stage:** Copies only the compiled binary into a minimal image

## 11. gRPC Protobufs

**Location:** `proto/`

### `inference.proto`

Defines the contract between the Go controller and the Python Inference Engine:

```protobuf
service InferenceService {
    rpc Predict(PredictRequest) returns (PredictResponse);
    rpc ReportMetrics(ReportMetricsRequest) returns (ReportMetricsResponse);
}
```

Key messages:
- `TaskInfo` — task name, image, command, dependencies, optional `ResourceHint`
- `ResourceHint` — `cpu_millicores` + `memory_mib` (avoids string parsing)
- `TaskPrediction` — predicted cpu/mem/duration + confidence score
- `TaskMetrics` — actual cpu/mem/duration + succeeded flag (for `ReportMetrics`)

### `optimizer.proto`

Defines the contract between the Go controller and the C++ Optimizer:

```protobuf
service OptimizerService {
    rpc Plan(PlanRequest) returns (PlanResponse);
}
```

Key messages:
- `DAGTask` — task spec with dependencies
- `TaskStatus` — current phase + allocated resources (map: taskName → status)
- `TaskPrediction` — estimated resources (map: taskName → prediction)
- `ResourceLimit` — `cpu_millicores` + `memory_mib` constraints
- `TaskAllocation` — which task, what cpu/mem request + limit to set

### `proto/gen/`

Pre-generated stubs — committed to the repo so no local `protoc` installation is needed:
- **`proto/gen/go/inference/`** — Go stubs for the controller to call Python
- **`proto/gen/go/optimizer/`** — Go stubs for the controller to call C++
- **`proto/gen/python/`** — Python stubs used by the inference server

*Note: C++ stubs are generated dynamically by CMake at build time into `optimizer-service/build/`.*

## 12. Web Dashboard (React UI)

**Location:** `ui/`

A **React + TypeScript** single-page app built with Vite. Served locally by `kwf ui` from `ui/dist/`.

### Key Features

1. **Visual Workflow Builder** — form-based UI to define tasks, images, commands, and dependencies without writing YAML
2. **Raw YAML Mode** — paste any `AdaptiveWorkflow` YAML directly
3. **"Compare Algorithms" Simulation** — an interactive 2D Tetris visualization comparing naive (Argo-style) vs. adaptive scheduling:
   - X-axis = time, Y-axis = memory usage
   - Baseline: tasks stacked at T=0, may exceed memory limit and crash
   - Adaptive: tasks slid rightward to stay below the memory ceiling
4. **Live Workflow Monitor** — polls `GET /api/workflows` every 3 seconds, shows all workflows with their live task phases as colored badges

### `src/App.tsx`

The main (and only) component. Key state:
- `workflows` — list fetched from `kwf ui` backend
- `tasks` — the list of task definitions in the Visual Builder
- `activeTab` — "builder" or "yaml"
- `showSimulation` — toggles the Tetris comparison modal

**`generateYamlFromState()`** converts the Visual Builder form state into a valid `AdaptiveWorkflow` YAML string with `apiVersion: v1.wannabe.dev/v1`.

**`calculateSimulationLayouts()`** runs two layout algorithms:
- **Baseline:** stacks tasks by start time, may pierce the memory limit line (shown in red)
- **Adaptive:** uses a bin-packing loop — pushes tasks right along the timeline until they fit below the limit

### API communication

All calls go to `http://localhost:8080` (the `kwf ui` backend). The backend proxies all REST calls to the real Kubernetes cluster via `controller-runtime` client.

## 13. PostgreSQL State Database

**Role:** Stores historical task execution metrics. Used by both:
- The **Go controller** → writes records when tasks complete (`StateDB.RecordExecution()`)
- The **Python Inference Engine** → reads records to make better predictions (`_get_history()`)

**Schema:** Single table `task_executions` with columns: `workflow_name`, `task_name`, `image`, `actual_cpu_m`, `actual_mem_mib`, `duration_s`, `succeeded`, `completed_at`.

**Why image-based?** The Inference Engine groups by container image. The assumption is: the same image will behave similarly every time it runs. This is a good heuristic for ML-style tasks (model training images, data processing images, etc.).

**Learning loop:**
```
Task completes → Controller writes to StateDB → Python reads from StateDB → Better predictions → Optimizer makes better plans → Tasks complete with fewer evictions
```

## 17. How Everything Connects

```
User writes test1-simple-linear.yaml
           │
           ▼
kwf submit test1-simple-linear.yaml
           │
           ▼  (k8s API call)
Kubernetes stores AdaptiveWorkflow CR
           │
           ▼ (controller watch event)
controller/Reconcile() called
           │
           ├─ Inference.Predict()
           │     ├─ [INFERENCE_GRPC_ADDR set] → HTTP/2 gRPC → Python server.py
           │     │       └─ ONNX model + Postgres history → per-task predictions
           │     └─ [not set] → baseline.go (copies BaseResources as-is)
           │
           ├─ Optimizer.Plan()
           │     ├─ [OPTIMIZER_GRPC_ADDR set] → HTTP/2 gRPC → C++ main.cpp
           │     │       └─ GreedyOptimizer::Plan() → which tasks to start
           │     └─ [not set] → greedy.go (same algorithm in Go)
           │
           ├─ createTaskPod() → Kubernetes creates Pods
           │
           └─ Status().Update() → updates TaskStatuses, phase

[10 seconds later]
controller/Reconcile() called again
           │
           └─ syncPodsToStatus() → reads Pod phases → updates TaskStatuses
                     │
                     └─ [on task success] StateDB.RecordExecution()
                               └─ PostgreSQL INSERT → feeds Python training data
```

## 18. Environment Variables

| Variable | Component | Default | Effect |
|----------|-----------|---------|--------|
| `INFERENCE_GRPC_ADDR` | Controller | _(unset)_ | `host:port` of Python gRPC server. If set, uses gRPC instead of in-process BaselineEngine |
| `OPTIMIZER_GRPC_ADDR` | Controller | _(unset)_ | `host:port` of C++ gRPC server. If set, uses gRPC instead of in-process GreedyOptimizer |
| `STATEDB_CONNECTION_STR` | Controller | _(unset)_ | PostgreSQL connection string. If set, writes execution metrics. Else, no-op |
| `ONNX_MODEL_PATH` | Python server | `model.onnx` | Path to trained ONNX model file |
| `DB_CONNECTION_STR` | Python server | _(unset)_ | PostgreSQL connection string for Python inference history queries |
| `GRPC_PORT` | Python server | `50051` | Port for the Python gRPC server |
| `GRPC_PORT` | C++ server | `50052` | Port for the C++ gRPC server |

## 19. Running Locally — Step by Step

### Minimal (no microservices — in-process only)

```bash
# 1. Start a Kind cluster
kind create cluster --name adaptive-dev

# 2. Install the CRD
make manifests
kubectl apply -f config/crd/bases/

# 3. Run the controller (uses in-process baseline + greedy optimizer)
make run

# 4. In another terminal, submit a test workflow
go run ./cmd/kwf submit test1-simple-linear.yaml

# 5. Watch status
go run ./cmd/kwf status test-linear-pipeline
```

### Full stack (with ML inference, C++ optimizer, and state DB)

```bash
# 1. Start all microservices
docker compose up -d

# 2. Start a Kind cluster  
kind create cluster --name adaptive-dev

# 3. Install CRD
make manifests
kubectl apply -f config/crd/bases/

# 4. Run the controller with all gRPC endpoints
INFERENCE_GRPC_ADDR=localhost:50051 \
OPTIMIZER_GRPC_ADDR=localhost:50052 \
STATEDB_CONNECTION_STR="postgres://adaptive:adaptive-dev-password@localhost:5432/adaptiveworkflows?sslmode=disable" \
make run

# 5. Open the web dashboard
go run ./cmd/kwf ui
# Then open http://localhost:8080

# 6. Or use the CLI
go run ./cmd/kwf submit test4-heavy-ml.yaml
go run ./cmd/kwf list
```

### Running tests

```bash
make test         # Unit tests + controller integration tests (envtest)
make lint         # Static analysis (golangci-lint)
make lint-fix     # Auto-fix linting issues
```
