# K8s Adaptive Workflows: File-by-File Documentation

This document explains exactly what each directory and major file does in the K8s Adaptive Workflows project.

## 1. Project Root

*   **`docker-compose.yml`**: The local development stack. Orchestrates the PostgreSQL database, the Python Inference Engine (`inference-engine`), and the C++ Optimizer Service (`optimizer-service`). The controller runs externally and connects to these containers via `localhost`.
*   **`Dockerfile`**: The multi-stage Dockerfile used to build the main Go Kubebuilder Operator (the Main Controller).
*   **`Makefile`**: The build automation script generated by Kubebuilder. Contains targets for code generation (`make generate`), CRD manifest generation (`make manifests`), testing (`make test`), and deployment (`make deploy`).
*   **`PROJECT`**: The Kubebuilder configuration file tracking the scaffolding layout, GVKs (Group/Version/Kind), and project domains. **Do not edit.**
*   **`go.mod` / `go.sum`**: The Go module files defining the dependencies for the controller, optimizer interface, and kwf CLI tools.

## 2. Main Controller & Operator (Go)

### `api/v1/`
*   **`adaptiveworkflow_types.go`**: Defines the `AdaptiveWorkflow` Custom Resource Definition (CRD). It contains the `Spec` (the goal, tasks, and constraints) and `Status` (task phases, resource usage) structs. Includes `+kubebuilder` marker comments to auto-generate Kubernetes manifest files.
*   **`groupversion_info.go`**: Sets up the API group (`v1.wannabe.dev`) and registers it with the Kubernetes scheme.
*   **`zz_generated.deepcopy.go`**: Auto-generated DeepCopy methods required for Kubernetes core types. **Do not edit.**

### `cmd/`
*   **`main.go`**: The entry point for the Kubernetes Operator. It initializes the `Manager`, sets up the `AdaptiveWorkflowReconciler`, connects to the gRPC services (if `INFERENCE_GRPC_ADDR` and `OPTIMIZER_GRPC_ADDR` are set), connects to the StateDB (if `STATEDB_CONNECTION_STR` is set), and starts the control loop.
*   **`kwf/main.go`**: The entry point for the `kwf` Command Line Interface (CLI). Built using Cobra, providing commands like `kwf submit`, `status`, and `list` for easy interaction with the custom resources.

### `internal/controller/`
*   **`adaptiveworkflow_controller.go`**: The heart of the operator. It contains the Reconcile loop which runs every time an `AdaptiveWorkflow` is created or updated. It is responsible for calling the Inference layer to get resource predictions, calling the Optimizer to get a scheduling plan, spawning Kubernetes Pods, updating resource statuses, and reporting back execution metrics to the StateDB upon task completion. 
*   **`adaptiveworkflow_controller_test.go`**: Unit tests verifying the reconcile loop logic (e.g., DAG ordering, resource constraints) using the Ginkgo/Gomega testing framework.

### `internal/inference/` & `internal/optimizer/`
*   **`engine.go` / `optimizer.go`**: The Go interfaces defining how the controller talks to these components.
*   **`baseline.go` / `greedy.go`**: The "in-process" fallback implementations. If gRPC environments are not set, the controller will default to these basic Go implementations.

### `internal/grpcclient/`
*   **`inference_client.go`**: A wrapper that implements `inference.Engine` but passes all traffic over gRPC to the standalone Python Inference Engine microservice.
*   **`optimizer_client.go`**: A wrapper that implements `optimizer.Optimizer` but passes all traffic over gRPC to the standalone C++ Optimizer microservice.

### `internal/statedb/`
*   **`client.go`**: Implements the `StateDB` interface using the `database/sql` PostgreSQL driver. Used to insert execution data (`TaskExecutions`) and query historical aggregations (`GetHistoryByImage`). Contains a `NoOpStateDB` fallback if the database URL isn't configured.
*   **`schema.sql`**: The PostgreSQL schema definition (creates the `task_executions` table). It is automatically mounted into the `postgres` container upon startup.

## 3. Python Inference Engine

**Directory:** `inference-engine/`

*   **`README.md`**: Instructions on how to build, run, and host the Python ML microservice.
*   **`server.py`**: A Python gRPC server. Implements `InferenceService` providing two methods:
    1.  `Predict`: Uses a loaded ONNX model to predict CPU/Memory boundaries. Plugs into the StateDB to calculate historical rolling averages.
    2.  `ReportMetrics`: Accepts completed task metrics and writes them to PostgreSQL.
*   **`train_model.py`**: A script to build a `RandomForestRegressor` based on synthetic historical data and export it to `model.onnx`. Automatically invoked whenever the `Dockerfile` is built.
*   **`Dockerfile`**: Builds the Inference Engine container. Also runs the `train_model.py` script so an ONNX model is baked into the runtime image.
*   **`requirements.txt`**: Python dependencies (`grpcio`, `onnxruntime`, `scikit-learn`, `psycopg2`).

## 4. C++ Optimizer Service

**Directory:** `optimizer-service/`

*   **`README.md`**: Build instructions, hosting setup, and testing commands for the C++ Optimizer microservice.
*   **`src/greedy_optimizer.h` / `src/greedy_optimizer.cpp`**: The core graph traversal algorithm. It checks dependency trees, determines "ready" tasks, compares predictions against max bounded resource constraints `MaxResources`, and generates a scheduling plan.
*   **`src/main.cpp`**: The C++ gRPC server entry point hooking into the generated `.grpc.pb.h` sources. Transforms protobuf request messages into C++ standard library equivalents before passing them to the greedy optimizer.
*   **`CMakeLists.txt`**: C++ build configuration script. Locates `gRPC` and `protoc`, auto-generates the C++ protobuf stubs, and creates the executable server binary.
*   **`Dockerfile`**: A multi-stage Docker build taking Ubuntu 22.04, retrieving the compiler toolchain, compiling the C++ `optimizer_server`, and then copying the compact executable into a minimal runtime image.

## 5. gRPC Protobufs & Stubs

### `proto/`
*   **`inference.proto`**: The gRPC protocol defining the messages for `PredictRequest`, `PredictResponse`, and `TaskMetrics`.
*   **`optimizer.proto`**: The gRPC protocol defining the messages for `PlanRequest`, `TaskStatus`, and `TaskAllocation` responses.

### `proto/gen/`
*   Contains the auto-generated stubs for language-specific interactions.
    *   **`go/`**: The Go server/client stubs used by the Controller.
    *   **`python/`**: The Python server stubs used by the Inference Engine.
    *   Note: The C++ stubs are auto-generated dynamically at build-time via CMake inside `optimizer-service/build/generated/`.

## 6. Manifests & Config

**Directory:** `config/`

*   **`crd/`**: Kubebuilder auto-generated Kubernetes YAML files defining the Custom Resource structure natively for Kubernetes.
*   **`rbac/`**: Kubebuilder auto-generated Kubernetes RBAC (Role-Based Access Control) permissions allowing the Go Controller to watch Pods and update AdaptiveWorkflows. 
*   **`samples/v1_v1_adaptiveworkflow.yaml`**: The main example schema (a mini ETL pipeline) used to test the project locally.
*   **`statedb/`**: Plain-text Kubernetes deployment manifests to host the PostgreSQL database inside the cluster (Includes Deployment, Service, Secret, PVC, and Kustomization linking to `schema.sql`).
